00. Submission Mode: Devnet Acceptable Fallback

Solstice is currently positioned as a Devnet working prototype with an explicit fallback path for infrastructure that only becomes fully meaningful on mainnet.

What is Proven in Devnet Mode

  • Real Devnet transaction construction, signing, submission, and finality tracking.
  • Fault-injected expired blockhash failures that move through the retry planner and queue.
  • Persistent lifecycle evidence with timestamps for processed, confirmed, and finalized states whenever stream data is available.
  • Dynamic tip calculation code and Jito bundle construction code wired into the same orchestrator path.
  • AI decision infrastructure with schema validation, audit logging, and deterministic fallback safeguards.
  • Yellowstone/Geyser streaming as the preferred slot source, with RPC WebSocket fallback.

What is Not Claimed in Devnet Mode

  • Devnet does not prove a real mainnet Jito bundle landing.
  • Direct RPC fallback records are marked as fallback execution records, not successful mainnet Jito bundle IDs.
  • Mainnet Jito landing proof is a separate final proof run using a capped funded wallet.

This separation is intentional. It keeps the prototype honest for judges while still allowing the complete stack to run end-to-end before the mainnet spend.

02. Core Components & Responsibilities

A

Yellowstone gRPC & WS Fallback

Subscribes to live slot changes from the Yellowstone Geyser gRPC daemon. If native bindings fail (such as the napi-rs compilation locks on Windows Node 24), it transparently downgrades to a standard JSON-RPC WebSocket listener (onSlotChange) so slot feeds remain uninterrupted.

B

Leader Cache & Scheduler

Caches the active epoch-level leader schedule via Helius/RPC. Correlates scheduled validator public keys against Jito-enabled validator nodes. Times transaction bundles to execute precisely during identified Jito leader slots.

C

Jito Bundle Engine & Tip floor API

Builds multi-transaction bundles and appends a dynamic tip transfer instruction targeting a random active Jito tip account. Queries Jito's REST API (bundles/tip_floor) every 60s to dynamically adjust tips based on live landing rates (p50 baseline up to p75 under load) without hardcoded values.

D

AI Decision Engine (Gemini 2.0)

Invoked upon poller timeouts or transaction drops. Receives a detailed prompt outlining upcoming leaders, gas/compute usage, blockhash errors, and recent Jito tip history. Renders structured retry plans parsed and validated with Zod. Includes a circuit breaker that defaults to rules-based logic during API outages or quota limits.

03. Submission & Recovery Flows

Happy Path Submission Flow

  1. API receives a transaction request, simulating it to extract exact Compute Units (CU).
  2. Bundle Constructor appends the dynamic tip floor and signs a Solana Versioned Transaction.
  3. Leader Scheduler checks upcoming slots and broadcasts the bundle during the next Jito leader slot.
  4. Yellowstone gRPC streaming (or RPC websocket polling) monitors slot finalization.
  5. The transaction reaches FINALIZED and pushes updates to the dashboard via WebSockets.

Expired Blockhash Fault Recovery Flow

  1. Fault Injection route (/api/v1/transactions/expired) submits a transaction signed with a stale genesis blockhash.
  2. Confirmation Poller attempts to confirm the signature. After 30 attempts without block inclusion, the transaction times out.
  3. The state transitions to FAILED, classified as BLOCKHASH_EXPIRED.
  4. The AI agent is triggered. It detects the expired blockhash, fetches a **fresh blockhash**, recalculates the Jito tip, and sets `shouldRetry: true`.
  5. A BullMQ retry job is dispatched to Upstash Redis with the calculated delay.
  6. The BullMQ worker executes, rebuilds/re-signs the transaction, and broadcasts it.
  7. The transaction lands successfully on attempt 2, transitioning to FINALIZED.

04. Infrastructure & State Storage

SQLite & Prisma ORM

Stores permanent logs of transactions, bundles, and ai_decisions. This solves the issue of dashboard data clearing on server restarts, reloading transaction histories on startup.

BullMQ & Upstash Redis

Manages background retry states, isolating failures and executing transaction re-submits with exponential backoff or dynamic delays suggested by the AI agent.

Prometheus & Pino

Pino captures JSON structured logs from backend services. prom-client compiles landing latencies, AI confidence rates, and priority fees, exposing them at /metrics.
